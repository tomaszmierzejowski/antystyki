# Grafana + Loki Observability Guide

This guide explains how our Serilog + Promtail + Loki + Grafana stack is configured and how to easily query production logs in the new setup.

## 1. How the Setup Works

1. **.NET Backend (Serilog)**: We use `Serilog.Formatting.Json.JsonFormatter` in the [appsettings.json](file:///c:/Users/tmier/source/repos/antystics/backend/Antystics.Api/appsettings.json). This ensures that all structured log properties (such as `Level`, `EventType`, `Component`, `Url`) and any exceptions are formatted into a clean, flat JSON string where keys are properly PascalCased.
2. **Promtail**: The Promtail container runs as a Docker container, mounting the host's `/var/lib/docker/containers` and listening to the Docker socket. It relies on the Docker pipeline stage to automatically pick up `stdout` output from the `backend` and `frontend` containers. 
3. **Loki & Grafana**: Promtail pushes these logs to Loki. Grafana connects to Loki as a data source. To parse and query the structured JSON from Serilog, Grafana dashboards use the `| json` filter in their LogQL queries.

## 2. LogQL Cheat Sheet

Now that all logs are properly formatted as JSON, you must pipe the log stream through the [json](file:///c:/Users/tmier/source/repos/antystics/backend/Antystics.Api/appsettings.json) parser to search specific properties.

### Basic Searches
* **Get all backend logs:**
  `{service="backend"}`
* **Get all frontend logs:**
  `{service="frontend"}`

### Filtering by JSON Properties (Case-Sensitive!)
After piping your log stream to `| json`, you can query against the specific properties that Serilog outputs. Since we use `JsonFormatter`, properties (even custom ones from string interpolation) are usually PascalCased.

* **Find all backend errors:**
  `{service="backend"} | json | Level="Error"`
* **Find a specific event type (e.g., failed logins):**
  `{service="backend"} | json | EventType="auth.failed_login"`
* **Find a specific Exception message containing a word:**
  `{service="backend"} | json | Level="Error" | Exception != "" |= "NullReferenceException"`
* **Find logs generated by a specific component (from [LogsController.cs](file:///c:/Users/tmier/source/repos/antystics/backend/Antystics.Api/Controllers/LogsController.cs)):**
  `{service="backend"} | json | Component="LoginForm"`

### Tracking Errors Over Time (Metrics from Logs)
To count occurrences over time (useful for dashboard charts):
* **Count backend errors per 5 minutes:**
  `sum(count_over_time({service="backend"} | json | Level="Error" [5m]))`
* **Count failed logins grouped by the 'Reason' property:**
  `sum by (Reason) (count_over_time({service="backend"} | json | EventType="auth.failed_login" [1h]))`

## 3. Best Practices for Developers

To ensure your logs populate the Grafana dashboards beautifully, follow these rules when writing `.NET` code:

1. **Never use string concatenation for logs.** Always use Serilog's structured syntax.
    * ❌ **Bad:** `_logger.LogError("Failed to sync user " + userId);`
    * ✅ **Good:** `_logger.LogError("Failed to sync user {UserId}", userId);`
2. **Use consistent Event Types.** If logging a significant domain event, explicitly log an `EventType` property. This makes alerting and dashboarding incredibly easy.
    * ✅ **Good:** `_logger.LogInformation("Order completed eventType={EventType} orderId={OrderId}", "orders.completed", order.Id);`
3. **Pass Exception objects explicitly.** When an exception is caught, pass it as the *first* argument to the logger so Grafana gets the full, multiline stack trace.
    * ❌ **Bad:** `_logger.LogError($"Error happened: {ex.Message}");`
    * ✅ **Good:** `_logger.LogError(ex, "Failed to calculate statistics for user {UserId}", userId);`
